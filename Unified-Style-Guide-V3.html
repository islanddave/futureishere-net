Mark-Done Button

Placed immediately after the Done-When box. First click = confirm complete (stamp + confetti fires, card settles green). Second click = undo (hover turns red to signal destructive). Always reversible. Text changes to reflect state — never relabel as "Edit" or "Change".

Compound Systems
Patterns
How individual components combine into larger interaction systems. Follow the assembly order — each layer depends on the one below it.
Progress & Completion System

A four-layer system that works together. Implement all four or none — partial implementation breaks the reward loop that makes it work for ADHD users.

Assembly order
1

Step checkboxes — micro-reward

One checkbox per numbered step. Checking = immediate visual feedback (checked state + strikethrough). Updates the card-level mini progress bar. State stored to localStorage key per card+step.

2

Mini progress bar — card momentum

Shows X/N steps within the card. Sits above the step list. When all steps are checked, the Mark-Done button label changes to "🎉 All steps done — mark section complete?" — a natural nudge, not a forced transition.

3

Mark-Done + stamp + confetti — section milestone

User clicks Mark-Done → stamp badge animates onto card header → localized confetti burst from button position (canvas-confetti, brand colors: sage/gold/cream/terra) → card settles: top bar turns success-green, trigger background tints success-pale, stamp persists. Global progress bar advances.

4

Global progress bar — full journey view

Always visible below sticky nav. As sections are marked done the sage→gold gradient fill advances. Label transitions: "X sections to go" → "Setup complete! 🎉" on final completion. Final completion triggers full-page bilateral confetti + celebration overlay (shown once per localStorage state, never again).

Why this works for ADHD: Each layer provides a reward at a different time scale — step (seconds), card (minutes), global (session). The system never requires completing a full card before getting feedback. Even one checked step is rewarded. The celebration overlay fires once and is never shown again, preserving its emotional weight.
Completed Card State

When a card is marked done, four visual changes apply simultaneously. The card remains fully accessible — open/close still works, all content remains visible. Completion is cosmetic and reversible.

localStorage Strategy

All progress state lives in a single versioned key (e.g. myguide_v1). Use a version suffix so schema changes don't corrupt existing user state — bump the suffix, old key is abandoned. Two sub-objects: steps (keyed cardId|stepNum) and cards (keyed cardId). Always wrap in try/catch — published artifact iframes may restrict storage access.

Always provide a Reset progress link — discreet, footer placement, behind a confirm() dialog. Users who accidentally mark things done need a recovery path, and power users re-running the guide appreciate the clean slate.

Rules
Non-negotiable constraints derived from production failures. Each rule has a specific failure mode that motivated it.
Rule 1 — In-page navigation in published artifacts
Never use <a href="#section"> for in-page scrolling in claude.ai published artifacts.

Failure mode: Published artifacts run inside a sandboxed <iframe>. The claude.ai parent page intercepts anchor click events that bubble up from inside that iframe. Any <a> click — even with a fragment-only href="#..." — is routed to the Claude prompt input instead of scrolling the page.

Fix — replace anchors with buttons
❌ Broken in published artifact
<a href="#learncube">
LearnCube
</a>
✅ Works in published artifact
<button onclick="document
.getElementById('learncube')
.scrollIntoView({behavior:
'smooth'})">
LearnCube
</button>
Note: href="https://..." with target="_blank" is unaffected — external links open correctly.
Rule 2 — JavaScript safety in embedded contexts

Iframe JS engines can use strict or non-standard parsing. A single syntax error in a <script> block kills the entire block — including unrelated functions like toggle() and openCard().

Three specific rules
1 · Always use catch(e) — never bare catch {}

Optional catch binding (ES2019) is rejected by some iframe JS engines as a syntax error.

try { ... } catch { }
// ❌ ES2019 — may break
try { ... } catch(e) { }
// ✅ ES5 — always safe
2 · Never put apostrophes inside single-quoted JS strings

An unescaped ' inside a single-quoted string terminates the string early and breaks the entire script block. Affects curly/smart apostrophes too.

'You're good to go.'
// ❌ apostrophe breaks string
'You are good to go.'
"You're good to go."
// ✅ rephrase or use double quotes
3 · Never use the implicit global event object

The implicit event object (used as event.target without passing it) is deprecated and unreliable in strict/sandbox contexts. Always pass the event or element explicitly.

function handle(type) {
event.target.classList
.add('active');
}

onclick="handle('x')"
// ❌ global event object
function handle(type, el) {
el.classList
.add('active');
}

onclick="handle('x', this)"
// ✅ explicit this
Rule 3 — Backslash-escaping in HTML attributes
Never use \" inside HTML attributes. HTML parsers do not recognise backslash as an escape character in attribute values. Only &quot; is valid.

This often happens when Python string formatting generates HTML with escaped quotes. The resulting onclick=\"fn()" in the raw file is silently mis-parsed — the onclick handler is dead with no console error. Use single-quoted attribute values and double-quoted JS strings, or audit with node --check on extracted JS + Python html.parser on the HTML before shipping.

Validation Checklist (before publishing)
□ Run node --check extracted.js on all inline script blocks — exit 0 required
□ Parse HTML with Python html.parser — zero unclosed tags, zero structural errors
□ Grep raw file for \" (backslash + double-quote) — zero allowed in attribute positions
□ Grep JS for bare catch { without binding — replace all with catch(e)
□ Grep for <a href="# in markup — replace every internal anchor with button onclick scrollIntoView
□ Grep JS for event.target not in an event handler parameter — replace with this passed explicitly
□ Scan all JS string literals containing apostrophes — rephrase or switch to double-quote delimiters
□ All getElementById targets in JS exist as id= in HTML
□ All onclick function names are defined in the script block
