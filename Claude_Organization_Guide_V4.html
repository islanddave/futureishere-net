🌱 Section 1 — what are these things?
💡
The Five Pieces of Claude
You only need two to get started
+
TL;DRProjects keep everything in one place so you never have to re-explain yourself. That's the main thing.
ThingWhat it actually is
💬 ChatsIndividual conversations. Like text messages — each one is its own thread.
📁 ProjectsFolders that keep related chats together, with shared memory and documents.
📚 Knowledge BaseThe files you upload to a project. Claude reads them in every chat automatically.
⚙️ Custom InstructionsRules you set once per project: "be concise", "use formal tone", etc. Always applied.
🧠 MemoriesLightweight facts Claude remembers across all your chats globally, like your name.
The only thing you really need to know:
Projects solve the most annoying problem — having to re-explain everything every time you open a new chat. Once set up, every chat starts already knowing your context.
🚀 Section 2 — your first project
🗂️
Making Your First Project
Three steps, then you're done
+
TL;DRClick Projects in the sidebar → New Project → give it a name. That's the whole thing. You can add documents and instructions later.
1
Open the sidebarIn Claude, look at the left sidebar and click Projects — or hover the left edge if the sidebar is hidden.
2
Click + New ProjectIt's in the upper right corner. Give it a name. Something obvious like "Work: Proposals" or "Hobby: Game Dev".
3
Hit create. You're done.The project exists and is already useful. Add documents and instructions when you're ready — not right now if you don't want to.
How to name your projects

Use a prefix so they sort cleanly and you can scan them quickly at a glance:

⚙️ WORK: prefix
  • WORK: Client Name
  • WORK: Proposals
  • WORK: Engineering
🎨 HOBBY: prefix
  • HOBBY: Game Dev
  • HOBBY: Writing
  • HOBBY: Music

Also create one called META: Shared Resources — this is your central library for templates and style guides you use across everything. More on this in Section 6.

Free accounts: 5 projects. Pro/Team ($20+/month): unlimited projects, plus smarter document retrieval that works 10× better with large files.
📚 Section 3 — adding your documents
📄
The Knowledge Base
Upload once, available forever in every chat
+
TL;DRDrag a file into your project. Claude reads it in every chat automatically from now on. No re-uploading. No re-explaining.

The Knowledge Base is the panel on the right side of a project. Anything you upload there is available to Claude automatically in every chat within that project.

Good things to upload
🎨
Style guides and tone documentsTell Claude once how you like things written. It'll remember in every chat.
📋
Templates you want Claude to matchUpload an example output you love. Ask Claude to match it.
📁
Background briefs and reference materialClient context, project background, technical docs — anything Claude needs to know.
✅
Past approved outputsThese are your best few-shot examples. Claude will use them as a reference for what "good" looks like.
Four rules that keep it working well
1
Name files clearly"Brand Voice Guide.pdf" is infinitely better than "doc1.pdf". Claude uses filenames to understand context.
2
One topic per fileFocused files retrieve more accurately than one giant catch-all document.
3
Test it worksAfter uploading, ask Claude: "What does my style guide say about tone?" — this confirms it can actually find the content.
4
Don't overload itOnly upload what's actually relevant. More isn't always better — irrelevant files can confuse retrieval.
Claude accepts PDF, DOCX, TXT, CSV, HTML, and most text-based formats. Total content fits roughly 150,000 words — paid plans expand this significantly.
⚙️ Section 4 — teaching Claude how you work
✍️
Custom Instructions
Write once, applied to every single chat
+
TL;DRWrite a few sentences about how you want Claude to behave. It follows them in every chat in that project, forever, without you repeating them.

Think of it like a one-time briefing for a new team member. Write it once; they remember it always.

What to include
👤
Who you are"You're helping a freelance designer who works with small business clients..." — give Claude your context.
🎙️
ToneFormal, casual, warm, concise, detailed. Be specific — "warm but professional" beats just "friendly".
📐
Format defaults"Always use headers. Keep bullet lists short. Never use filler phrases."
🚫
Constraints"Don't suggest tools outside our approved stack." What you don't want matters as much as what you do.
📄
What "document" means to you"When I say document I mean a .docx file at US Letter size." Defaults save a lot of back-and-forth.
✏️ Writing tip
Negatives are often more useful than positives. Instead of "be clear" try "never use jargon without explaining it". Instead of "be helpful" try "don't add advice I didn't ask for". Test your instructions with a few prompts and adjust — they rarely nail it on the first try, and that's completely normal.
Memories vs. Project Instructions — the easy rule
🧠 Memories (global)
  • Your name
  • Universal tone preferences
  • Formatting habits you want everywhere
⚙️ Project Instructions
  • Domain-specific rules
  • Role context for that project
  • Project-specific constraints

Don't put the same thing in both places. Global preferences go in Memories; project-specific rules go in custom instructions.

🗂️ Section 5 — separating work and hobby
🔀
Work Projects vs Hobby Projects
They don't bleed into each other — that's the whole system
+
TL;DROne project per domain. Work projects get formal instructions. Hobby projects get personal ones. They don't influence each other unless you deliberately connect them.
⚙️ Work projects
  • Client briefs and background docs
  • Brand and style guides
  • Process documentation
  • Past approved outputs
  • Formal, precise tone
  • Role context in instructions
🎨 Hobby projects
  • Reference material, inspiration
  • Personal style notes
  • Example outputs you like
  • Creative or world-building docs
  • Casual, exploratory tone
  • Open-ended mode encouraged
Keep sensitive client information in work-only projects. Work and hobby projects are completely isolated from each other. Data you share in a project is not used to train Claude's models without your explicit consent.

First: The META Project

Create one project called META: Shared Resources. This is your central library — not for chatting, just for storing assets you'll reference everywhere:

🎨
Your master style guideThe definitive document for how your writing should look and sound.
📋
Document templatesCover page layouts, memo structures, proposal frameworks.
✅
Approved example outputsYour best past work that you want future work to match.
The snapshotting workflow
1
Generate something you loveProduce the document or design you're happy with.
2
Extract the templateAsk Claude: "Turn this into a reusable template with placeholder text for project-specific content."
3
Name it with a version"Proposal Template v1 — Mar 2026.docx" — date and version number, every time.
4
Upload to METANow it's available to attach in any future chat in any project.
5
Never overwriteKeep old versions. You may want to go back someday. Storage is cheap, regret is not.
Snapshotting your style guide — the most useful one
🎨 Try this prompt when you're ready
"Summarize the writing style, formatting preferences, and tone rules we've been using in this project into a reusable style guide document I can save and apply to other projects."

Review the output, adjust it, then save as "Style Guide — [Project Name] — v1.md" and upload to META: Shared Resources.

🔀 Section 7 — using docs across projects
🔗
Cross-Project References
How to use a style guide from one project in another
+
TL;DRClaude can't see Project A while you're in Project B. But you can easily bring any document into a chat by attaching it. Three ways to do this.
⏭️
Safe to skip for now
Only relevant when you want to apply a style guide from one project while working in a different one. If you're not doing that yet, come back when you are.
Claude can only see what's in the current chat's context window. If a document lives in Project A and you're chatting in Project B, Claude can't see it automatically. You have to bring it in.
Option A — Project Knowledge Base
⭐
Best for documents you need all the timeUpload to that project's knowledge base. Available in every chat in that project automatically. This is the right choice for your style guide, templates, and regular reference docs.
Option B — Attach in Chat
📎
Best for one-off usesUpload the file directly into a specific chat message. It's available for that chat session only — doesn't clutter the knowledge base permanently.
Option C — The Cross-Project Prompt

When you want to use a style from another project, use this exact pattern:

📋 The template prompt
"Create a document about [[[subject and description]]] and use the style guide from the project called [[[project name]]]."
💬 Real example
"Create a document about our Q2 product roadmap (focus: mobile features, three phases) and use the style guide from the project called WORK: Brand Standards."

For this to work, the style guide must also be present in the current chat. Your three options:

Upload it to this project's knowledge base Attach it directly in this chat message Paste a summary into the project's custom instructions
💬 Section 8 — everyday chat habits
💬
Everyday Chat Habits
Small habits that save a lot of friction later
+
TL;DRStart a new chat when switching tasks. Write a descriptive first message. Archive anything important by asking Claude to summarize it.
Think of it this way: A project is a room. A chat is a meeting that happens in it. You can have many chats in one project — they each inherit the project's context but don't interfere with each other.
When to start a new chat
🔄
When switching to a different sub-taskDifferent tasks deserve their own thread. It keeps context clean and focused.
📏
When a chat has gotten very longVery long threads can get a bit muddled. A fresh chat with the important context pasted in is often cleaner.
🌱
When you want a fresh startSometimes you want Claude to approach something without the weight of the prior conversation.
Descriptive first messages matter

Your first message in a chat becomes its title in the sidebar. Make it descriptive and you'll be able to find it again later.

😬 Hard to find later
  • "hey"
  • "help me"
  • "continue from before"
✅ Easy to find later
  • "Draft Q3 roadmap email to stakeholders"
  • "Revise chapter 3 opening — more tension"
  • "Proposal for Acme website refresh"
Saving a valuable chat
🗂️ Archive a chat's key outputs
"Summarize the key outputs and decisions from this conversation into a structured document I can save and reference later."

Save or export the result, then upload to your project knowledge base so future chats can reference it.

🌿
A Few More Things Worth Knowing
When you've got the basics down
+
Version your documents

Claude regenerates things slightly differently each time. If an output matters, save it immediately. Use: DocumentName — vN — YYYY-MM.ext. Upload approved versions to the project knowledge base so you can reference them explicitly in future chats.

Iterate your custom instructions

Your instructions will get better over time. After a few sessions, note what Claude got wrong and adjust. Keep a "v1", "v2" archive of your instructions so you can roll back if a change makes things worse.

When NOT to use a project
✨
Use a regular chat for...One-off questions, quick experiments, and tasks where you'll never need the context again. Projects add value when you need persistence and consistency. For everything else, a regular chat is faster.
