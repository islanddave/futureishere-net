§1 — The Five Pieces of Claude
You only need two to get started
+
TL;DRProjects keep everything in one place so you never have to re-explain yourself. That's the main thing worth knowing right now.
ThingWhat it actually is
💬 ChatsIndividual conversations. Each one is its own thread — like a separate text message conversation.
📁 ProjectsFolders that keep related chats together, with shared memory and documents.
📚 Knowledge BaseThe files you upload to a project. Claude reads them in every chat automatically — no re-uploading.
⚙️ Custom InstructionsRules you write once per project: "be concise", "use formal tone", etc. Always applied.
🧠 MemoriesLightweight facts Claude remembers across all your chats globally, like your name and preferences.
The key insight: Without projects, every chat is a cold start — you re-upload the same files and re-explain the same context. Projects fix this. Every chat inside a project already knows your context. You just talk.

§2 — Making Your First Project
Three steps and you're done
+
TL;DRClick Projects in the sidebar → New Project → give it a name. That's the whole thing. Add documents and instructions when you're ready — not right now if you don't want to.
1
Open the sidebarIn Claude, look at the left sidebar and click Projects — or hover the left edge if the sidebar is hidden.
2
Click + New ProjectIt's in the upper right corner. Give it a name. Something obvious like "Work: Proposals" or "Hobby: Game Dev".
3
Hit create. You're done.The project exists and is already useful. Add documents and instructions when you're ready — not right now if you don't want to.
How to name your projects

Use a prefix so they sort cleanly and you can scan them quickly:

⚙️ WORK: prefix
  • WORK: Client Name
  • WORK: Proposals
  • WORK: Engineering
🎨 HOBBY: prefix
  • HOBBY: Game Dev
  • HOBBY: Writing
  • HOBBY: Music

Also create one called META: Shared Resources — your central library for templates and style guides you'll use across everything. More in §6.

Free accounts: 5 projects. Pro/Team ($20+/month): unlimited projects, plus smarter document retrieval that works roughly 10× better with large files.

§3 — Adding Your Documents
Upload once, available in every chat forever
+
TL;DRDrag a file into your project. Claude reads it automatically in every chat from now on. No re-uploading. No re-explaining. It's just there.
Good things to upload
🎨
Style guides and tone documentsTell Claude once how you like things written. It remembers in every chat.
📋
Templates you want Claude to matchUpload an example output you love. Claude will use it as a model for what "good" looks like.
📁
Background briefs and reference materialClient context, project background, technical docs — anything Claude needs to know about your world.
✅
Past approved outputs as examplesYour best past work gives Claude a concrete target to aim for.
Four rules that keep it working well
1
Name files clearly"Brand Voice Guide.pdf" is infinitely better than "doc1.pdf". Claude uses filenames to understand context.
2
One topic per fileFocused files retrieve more accurately than one giant catch-all document.
3
Test it worksAfter uploading, ask Claude: "What does my style guide say about tone?" — this confirms it can find the content.
4
Don't overload itOnly upload what's actually relevant. More isn't always better — irrelevant files reduce retrieval accuracy.
Accepted formats: PDF, DOCX, TXT, CSV, HTML, and most text-based files. Total capacity is roughly 150,000 words — paid plans expand this significantly via smarter retrieval.

§4 — Custom Instructions
Write once, applied to every single chat
+
TL;DRWrite a few sentences about how you want Claude to behave in this project. It follows them every time, without you repeating them. Treat it like onboarding a new team member.
What to include
👤
Who you are"You're helping a freelance designer who works with small business clients..." — give Claude your context.
🎙️
ToneFormal, casual, warm, concise, detailed. Be specific — "warm but professional" beats just "friendly".
📐
Format defaults"Always use headers. Keep bullet lists short. Never use filler phrases."
🚫
Constraints"Don't suggest tools outside our approved stack." What you don't want matters as much as what you do.
📄
Defaults for common words"When I say document I mean a .docx file at US Letter size." Saves back-and-forth every time.
✏️ Writing tip that actually helps
Negatives are often more useful than positives. Instead of "be clear" try "never use jargon without explaining it". Instead of "be helpful" try "don't add advice I didn't ask for". Test with a few prompts and adjust — instructions rarely nail it first try, and that's completely normal.
Memories vs. Project Instructions — the easy rule
🧠 Memories (global)
  • Your name
  • Universal tone preferences
  • Formatting habits you want everywhere
⚙️ Project Instructions
  • Domain-specific rules
  • Role context for this project
  • Project-specific constraints

Don't put the same thing in both places. Global preferences → Memories. Project-specific rules → Custom Instructions.

§5 — Work Projects vs Hobby Projects
They don't bleed into each other — that's the whole system
+
TL;DROne project per domain. Work projects get formal documents and instructions. Hobby projects get personal ones. They don't influence each other unless you deliberately connect them (see §6 and §7).
⚙️ Work projects
  • Client briefs and background docs
  • Brand and style guides
  • Process documentation
  • Past approved outputs
  • Formal, precise tone
  • Role context in instructions
🎨 Hobby projects
  • Reference material, inspiration
  • Personal style notes
  • Example outputs you like
  • Creative or world-building docs
  • Casual, exploratory tone
  • Open-ended mode encouraged
Keep sensitive client information in work-only projects. Data you share in a project is not used to train Claude's models without your explicit consent.

First: The META Project

Create one project called META: Shared Resources. This isn't for chatting — it's your central library. Upload here:

🎨
Your master style guideThe definitive document for how your writing should look and sound.
📋
Document templatesCover page layouts, memo structures, proposal frameworks.
✅
Approved example outputsYour best past work that you want future work to match.
The snapshotting workflow
1
Generate something you loveProduce the document or design you're happy with in any project.
2
Extract the templateAsk Claude: "Turn this into a reusable template with placeholder text for project-specific content."
3
Name it with a version"Proposal Template v1 — Mar 2026.docx" — date and version number, every time.
4
Upload to METANow it's available to attach in any future chat in any project.
5
Never overwrite old versionsKeep old files. You may want to go back. Storage is cheap; regret is not.
Snapshotting your style guide — the most useful one
🎨 Try this prompt when you're ready
"Summarize the writing style, formatting preferences, and tone rules we've been using in this project into a reusable style guide document I can save and apply to other projects."

Review the output, adjust it, save as "Style Guide — [Project Name] — v1.md", and upload to META: Shared Resources.

§7 — Cross-Project References
How to use a style guide from one project in another
+
TL;DRClaude can't automatically see documents from Project A while you're in Project B. But you can bring any document into a chat by attaching it. Three ways to do this below.
⏭️
Safe to skip for now
Only relevant when you want to apply a style guide from one project while working in a different one.
↑ Back to top when you're ready to come back
Claude can only see what's in the current chat's context window. If a document is in Project A and you're in Project B, Claude can't see it. You have to bring the document in.
Option A — Project Knowledge Base
⭐
Best for documents you need all the timeUpload to that project's knowledge base. Available in every chat automatically. Right choice for your style guide, templates, and regular reference docs.
Option B — Attach in Chat
📎
Best for one-off usesUpload the file directly into a specific chat message. Available for that session only — doesn't clutter the knowledge base permanently.
Option C — The Cross-Project Prompt

When you want to use a style from another project, this is the prompt pattern to use:

📋 The template prompt
"Create a document about [[[subject and description]]] and use the style guide from the project called [[[project name]]]."
💬 Real example
"Create a document about our Q2 product roadmap (focus: mobile features, three phases) and use the style guide from the project called WORK: Brand Standards."

For this to work, the style guide must also be present in the current chat. Your three options:

Upload it to this project's knowledge base Attach it directly in this chat message Paste a summary into the project's custom instructions

§8 — Everyday Chat Habits
Small habits that save a lot of friction later
+
TL;DRStart a new chat when switching tasks. Write a descriptive first message. Archive anything important by asking Claude to summarize it.
Think of it this way: A project is a room. A chat is a meeting that happens in it. Many chats can live in one project — they each inherit the project's context but don't interfere with each other.
When to start a new chat
🔄
When switching to a different sub-taskDifferent tasks deserve their own thread. It keeps context clean and focused.
📏
When a chat has gotten very longVery long threads can get muddled. A fresh chat with the key context pasted in is often cleaner.
🌱
When you want a fresh startSometimes you want Claude to approach something without the weight of the prior conversation.
Descriptive first messages matter

Your first message in a chat becomes its title in the sidebar. Make it descriptive and you'll find it again easily.

😬 Hard to find later
  • "hey"
  • "help me"
  • "continue from before"
✅ Easy to find later
  • "Draft Q3 roadmap email to stakeholders"
  • "Revise chapter 3 opening — more tension"
  • "Proposal for Acme website refresh"
Saving a valuable chat
🗂️ Archive a chat's key outputs
"Summarize the key outputs and decisions from this conversation into a structured document I can save and reference later."

Save or export the result, then upload to your project knowledge base so future chats can reference it.

§9 — A Few More Things Worth Knowing
When you've got the basics down
+
Version your documents

Claude regenerates things slightly differently each time. If an output matters, save it immediately. Use: DocumentName — vN — YYYY-MM.ext. Upload approved versions to your project knowledge base so you can reference them explicitly in future chats.

Iterate your custom instructions

Your instructions will get better over time. After a few sessions, note what Claude got wrong and adjust. Keep a "v1", "v2" archive so you can roll back if a change makes things worse.

When NOT to use a project
✨
Use a regular chat for...One-off questions, quick experiments, tasks where you'll never need the context again. Projects add value when you need persistence and consistency. For everything else, a regular chat is faster and simpler.
